<threat_model>
  <threats>
    <threat severity="high">
      Client-controlled blackout bypass: The teacher form sends is_blackout as a form field
      value. Any client-side manipulation (DevTools form edit, direct fetch, Burp Suite
      interception) can set is_blackout=false for dates that are actually in the blackout table.
      The server currently uses this value verbatim to set status='pending', allowing the
      request to reach admins for manual review instead of being auto-denied. The teacher
      effectively bypasses the blackout enforcement by lying about the blackout status.
    </threat>
  </threats>
  <mitigations>
    <mitigation>
      Server-side override: Before computing status and before inserting the row, the server
      queries the blackout_dates table for any row where start_date <= request end_date AND
      end_date >= request start_date. This is the standard interval overlap condition. The
      result (serverBlackout: boolean) overrides is_blackout from the form. The insert uses
      serverBlackout for both the status field and the is_blackout column, so the DB reflects
      what the server determined — not what the teacher claimed.
    </mitigation>
    <mitigation>
      Blackout check error handling: If the Supabase query fails (network issue, misconfigured
      RLS, etc.), the action returns an error response rather than silently proceeding with the
      client-supplied value. Fail closed: a blackout check failure does not let requests through.
    </mitigation>
  </mitigations>
</threat_model>
